2. Core Concept (1 Paragraph)
SplitStream lets users create “pools” for shared subscriptions (Netflix, Spotify, etc.). One user is the
Host who sets up the pool. Others join as Contributors and pay their share via Stripe test payments.
A Trust Score tracks payment reliability. Hosts are protected by automated retry logic and contributor
replacement.

6. Stripe Payment Flow
6.1 T est Mode Setup
1. Create Stripe account (free, instant)
2. Get test keys from Dashboard → Developers → API Keys
3. Use test card: 4242 4242 4242 4242 (always succeeds)
4. Use test card: 4000 0000 0000 0002 (always fails)
6.2 Contributor Payment Flow
1. Contributor clicks "Pay My Share" on pool page
2. Frontend calls POST /api/payments/intent
Backend:
- Creates Stripe PaymentIntent (amount = pool.shareAmount)
- Saves Transaction record with status: 'pending'
- Returns client_secret to frontend
3. Frontend uses Stripe.js to confirm card payment
4. Stripe redirects back or sends webhook
5. Webhook 'payment_intent.succeeded' arrives:
- Find Transaction by stripePaymentIntentId
- Update status to 'completed'
- Update User trustScore: +2
- Update Pool: check if all contributors paid
6. Webhook 'payment_intent.payment_failed' arrives:
- Update Transaction status to 'failed'
- Update User trustScore: -10
- Schedule retry (node-cron, next day)
6.3 Host Payout (Mocked)
Trigger: All contributor transactions for cycle are 'completed'
1. Calculate: totalCollected - platformFee (5%)
2. Create Transaction: type='host_payout', status='completed' (mocked)
3. Update Pool: nextBillingDate += 1 month
4. In real app: this would be Stripe Connect transfer
6.4 W ebhook Events Handled
Event Action
payment_intent.succeeded Mark transaction completed, +2 trust
payment_intent.payment_failed Mark failed, -10 trust, schedule retry
charge.refunded Mark transaction refunded

18. Interview Talking Points
“Why MongoDB over PostgreSQL?”
“For this project, MongoDB’s flexible schema made sense for rapid iteration. Pools have dynamic
member arrays, and trust scores are derived from transaction history. For a production fintech app,
I’d use PostgreSQL with proper ACID transactions. ”
“How do you handle race conditions when two users join the last slot?”
“I use MongoDB’s atomic operations. The join operation checks members.length < slots in the
query condition, so only one succeeds. The other gets a ‘pool full’ error. ”
“What about idempotency in payments?”
“Each PaymentIntent has a unique ID stored in the Transaction collection. Before creating a new
charge, I check if a completed transaction exists for this user + pool + billing cycle. ”
“How would you scale this?”
“Add Redis for session caching, move cron jobs to a separate worker process, use Stripe Connect for
real payouts, add read replicas for MongoDB, and implement proper event-driven architecture with
a message queue. ”
